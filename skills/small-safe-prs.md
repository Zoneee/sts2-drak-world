# Skill：小而安全的 PR

## 目的
通过保持改动小巧、可评审、可回滚，提高吞吐量并降低风险。

## 规则
1. 一个 PR 只解决一个问题。
2. 不混合重构 + 功能 + 无关清理，除非有明确说明。
3. 优先增量交付，避免大范围重写。
4. 若任务较大，按以下顺序拆分：
   - 脚手架
   - 实现
   - 验证
   - 清理
5. PR 描述需明确说明范围和非范围。

## 经验阈值
- 尽量控制在 ~300 行改动以内
- 尽量控制在 ~10 个文件以内
- 超出阈值的 PR 需提供明确理由

## PR 必备章节
- 改动内容
- 改动原因
- 如何验证
- 风险
- 后续事项
