# Skill：先计划，后编码

## 目的
在实现前强制做出明确计划，防止失控改动。

## 适用场景
- 任何预计超过 15 分钟的任务
- 涉及多个文件或模块的改动
- 涉及 Schema、API、迁移、认证、基础设施或 UI 行为的工作
- 没有明显根因的 bug 修复

## 所需输入
- 任务描述
- 受影响的用户流程或 bug 报告
- 相关产品规格
- 相关架构文档
- 相关开发规范

## 执行步骤
1. 用 3-5 条要点总结问题。
2. 识别可能受影响的模块和文件。
3. 识别约束和禁止区域。
4. 定义验收标准。
5. 定义验证步骤。
6. 记录风险与未知项。
7. 较大的工作保存计划至 `docs/exec-plans/active/`，较小的工作在 PR/任务中内联记录。

## 输出格式
- 问题描述
- 范围
- 受影响区域
- 验收标准
- 验证计划
- 风险
- 后续事项
